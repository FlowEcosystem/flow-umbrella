# Umbrella — Архитектура

Last updated: 2026-04-20
Status: Draft v1

## Обзор

Umbrella — self-hosted платформа управления endpoint'ами, развёрнутая в нескольких физических локациях (филиалах) одной организации. Делает три вещи:

1. Централизованная фильтрация трафика на endpoint'ах (блокировки на уровне DNS/URL/портов).
2. Удалённое управление endpoint'ами (раздача политик, команды, телеметрия).
3. Агрегация и контроль по всем филиалам из центрального офиса (HQ).

Целевой масштаб: 180+ филиалов, 100–500 endpoint'ов на филиал (~35 000 endpoint'ов суммарно).

Umbrella позиционируется как self-hosted альтернатива Cisco Umbrella + Microsoft Intune, заточенная под организации с распределёнными физическими локациями, которым нужна автономия на каждой точке при единой оркестрации сверху.

## Топология

Umbrella — **федеративная, не multi-tenant.** Каждый филиал запускает свой автономный стек. HQ — отдельный сервис поверх всех филиалов, агрегирует данные. Филиалы не делят между собой инфраструктуру, и HQ не является точкой отказа для работы филиала.

```
                    ┌────────────────────────┐
                    │   HQ Server            │
                    │   hq.umbrella.su       │
                    │   (публичный IP)       │
                    │                        │
                    │   - глобальные политики│
                    │   - агрегированные     │
                    │     метрики            │
                    │   - реестр филиалов    │
                    │   - оркестровка        │
                    │     rollout'ов         │
                    └────────────▲───────────┘
                                 │
                   исходящие HTTPS от филиалов
                   (heartbeat, события, pull)
                                 │
      ┌──────────────┬───────────┼───────────┬──────────────┐
      │              │           │           │              │
┌─────┴────┐   ┌─────┴────┐  ...больше... ┌──┴───────┐  ┌───┴──────┐
│ Филиал 1 │   │ Филиал 2 │               │ Филиал N │  │Филиал 180│
│ (Казань) │   │ (СПб)    │               │          │  │          │
│          │   │          │               │          │  │          │
│ umbrella-server + локальная БД + админка │         │  │          │
│    ▲           ▲                         │         │  │          │
│    │ mTLS      │ mTLS                    │         │  │          │
│    │           │                         │         │  │          │
│ spider      agents (~100–500 хостов)     │         │  │          │
│ (роутер)                                 │         │  │          │
└──────────┘   └──────────┘                └─────────┘  └──────────┘
```

**Правила коннектов:**

- Филиалы инициируют весь исходящий трафик в HQ. HQ никогда не ходит в филиалы сам. Филиалы за NAT/файерволом, без публичных IP.
- Агенты и spider'ы подключаются только к локальному серверу своего филиала, никогда напрямую к HQ.
- HQ имеет статический публичный домен (`hq.umbrella.su`) и TLS-сертификат.
- У каждого филиала свой домен/DNS (например, `kazan.umbrella.su` или только локальный адрес).

**Автономность:**

- Если HQ недоступен — филиалы продолжают работать. Агенты опрашивают локальный сервер филиала, политики применяются, телеметрия собирается.
- Простой HQ отображается в админке филиала как предупреждение, не как блокер.
- Когда HQ возвращается — филиалы досылают накопившиеся события и подтягивают версии политик.

## Компоненты

### umbrella-server (филиал)

Python 3.13 / FastAPI / SQLAlchemy async / PostgreSQL 16. Упакован в Docker Compose-стек. По одному инстансу на филиал, по одной PostgreSQL-базе на филиал. Между филиалами нет общего хранилища.

Отвечает за:
- Админку для локального IT-персонала филиала.
- Enrollment агентов, heartbeat, раздачу политик, диспатч команд, сбор метрик.
- Зеркалирование глобальных политик из HQ; владеет локальными политиками.
- Периодическую исходящую синхронизацию с HQ (heartbeat, агрегированные метрики, критичные события, pull версий политик).

### umbrella-agent

Сервис на Go, ставится на Windows endpoint'ы (первая очередь), позже Linux/macOS. Windows-инсталлер — подписанный MSI; пакетирование для Linux через nfpm (deb/rpm).

Отвечает за:
- mTLS-аутентификацию против локального сервера филиала.
- Периодический опрос: политики (~60с), очередь команд (~15с), push метрик (~60с).
- Enforcement: блокировка на уровне DNS (конкретный механизм TBD — см. "Открытые вопросы").
- Исполнение whitelisted-команд с проверкой криптоподписи.
- Самообновление через команду `update_self`.

### umbrella-spider

Сервис на Go, ставится на Linux-шлюзы. Целевая платформа: Linux-роутер (Debian/Ubuntu), возможно OpenWrt позже. Mikrotik RouterOS не целевой — Go-бинари на нём нативно не запускаются.

Отвечает за:
- Enforcement на уровне сети через nftables/iptables или dnsmasq.
- Прозрачную фильтрацию для хостов за роутером (покрывает устройства без агентов — гостевые сети, BYOD).
- Тот же sync-протокол, что у агентов: опрос сервера филиала за правилами, push статистики трафика.

### umbrella-hq (в будущем)

Python/FastAPI-сервис, отдельный от сервера филиала. Переиспользует общий код (core, db, DI, capability-система) через shared-пакет в монорепе.

Отвечает за:
- Реестр филиалов: список, статус здоровья, количество агентов.
- Создание глобальных политик: политики, применяемые ко всем или выбранным филиалам.
- Агрегированные дашборды и отчёты.
- Оркестровку rollout'ов: раздача команд "обновитесь до версии X" филиалам пакетами.
- HQ-админов (отдельно от локальных админов филиалов).

### umbrella-web

SPA на Vue 3 / Vite / PrimeVue. Отдаётся статикой, ходит в API сервера филиала. Один build-артефакт используется всеми филиалами; для HQ будет отдельный build (или тот же с runtime-детектом API endpoint, TBD).

## Аутентификация и авторизация

Три независимых auth-контекста, никогда не смешиваются:

### Локальные админы филиала

- Учётка: email + пароль, хранится в БД филиала.
- Токены: JWT access (60 мин) + случайный refresh (14 дней) в httpOnly-cookie.
- Роли: `superadmin`, `admin`, `viewer`. Мапятся в capabilities через статическую таблицу (`domains/auth/capabilities.py`).
- Скоуп: только тот филиал, где админ был создан. Никакого кросс-филиального доступа.

### Агенты

- Учётка: mTLS клиентский сертификат, выпущенный внутренним CA филиала при enrollment'е.
- Enrollment: одноразовый токен, сгенеренный админом филиала, используется агентом в первом запросе `/v1/agent/enroll`. Агент локально генерирует приватный ключ и отправляет CSR; филиал подписывает.
- Ротация сертификата: короткий TTL (7–30 дней), агент обновляет до истечения.
- Скомпрометированный агент: сертификат отзывается через CRL; агент больше не подключится.

### HQ-админы (в будущем)

- Отдельная таблица пользователей в БД HQ. Никакого общего идентити с админами филиалов.
- Возможность добавить SSO (Microsoft 365, Google Workspace) позже.

### Филиал ↔ HQ

- Каждый филиал разово регистрируется в HQ, получает long-lived access token.
- Токен хранится на сервере филиала, используется для всех запросов к HQ.
- Ротация — ручная (пересоздать токен в HQ-админке → обновить конфиг филиала).

## Политики

### Поле source

У каждой политики есть поле `source`:

- `local` — создана админом филиала в локальной админке. Полностью редактируема на филиале. Не покидает филиал.
- `global` — создана в HQ, синхронизирована на филиалы. **На филиале read-only.** Удаление отражается через обновление версии политики в HQ.

### Разрешение конфликтов

Когда локальная и глобальная политики пересекаются по одному ресурсу (домен, категория, агент) — **по умолчанию побеждает global**. Филиалы могут добавить **более строгие** локальные политики, но не могут ослабить глобальные.

Opt-in более слабое переопределение: HQ может пометить глобальную политику как `overridable=true`, тогда локальные политики могут её перебить. Используется для рекомендательных-но-гибких настроек по умолчанию.

### Версионирование

Глобальные политики версионируются. Протокол синхронизации:

```
филиал → HQ: GET /v1/branches/policies?since_version=123
HQ    → филиал: [политики с version > 123, + текущая версия]
филиал применяет идемпотентно по hq_policy_id, сохраняет применённую версию
```

Это позволяет филиалам, которые были оффлайн, догнать после реконнекта.

## Команды

Remote command execution — это фича с серьёзными последствиями для безопасности. Фазовая раскатка и строгие ограничения обязательны.

### Фазы

**Фаза 1 — безопасные команды (пилот):**

- `reboot` — плановая перезагрузка с уведомлением пользователя.
- `collect_diagnostics` — агент собирает логи и информацию о системе, заливает на сервер для скачивания админом.
- `update_self` — агент скачивает новый бинарник и перезапускается.
- `apply_config` — агент сразу перечитывает конфиг, не ждёт следующего poll'а.

Команды Фазы 1 — низкорисковые, покрывают 80% операционных задач (жизненный цикл агента, диагностика).

**Фаза 2 — опасные команды (после пилота):**

- `download_file` — скачать файл из whitelisted-источника в указанную папку.
- `run_installer` — запустить MSI из подписанной серверной локации.
- `disable_network_interface` — переключить сетевой адаптер.
- `run_named_script` — выполнить заранее зарегистрированный скрипт по имени (не произвольный код).

Каждый тип команды Фазы 2 добавляется по отдельности, с типизированными параметрами и аудит-логом. Произвольное исполнение shell **никогда** не поддерживается.

**Фаза 3 — кампании и расписание:**

- Пакетные rollout'ы с лимитом параллельности и возможностью abort'а.
- Команды по расписанию (cron-подобно).
- Условные команды (запустить только если версия агента попадает под предикат).

### Подпись

Все отправляемые команды криптографически подписываются сервером филиала приватным ключом, который хранится вне БД (`/etc/umbrella/command-signing.key`, права 600). Агенты проверяют подпись соответствующим публичным ключом, полученным при enrollment'е.

Одной компрометации БД недостаточно для подделки команд — атакующему нужен также доступ к файловой системе, который защищён правами ОС.

### Ограничение на стороне агента

Агенты держат в коде захардкоженный whitelist типов команд, которые могут исполнять. Даже подписанная команда неизвестного типа отвергается. Это defense in depth: даже скомпрометированный сервер не может приказать агентам сделать произвольное.

### Approval flow

Отложено до решения по Фазе 2. Допущение: команды Фазы 1 — self-service одним админом; Фаза 2 вводит требования апрува для деструктивных операций.

### Аудит

Каждая отправка и результат команды логируется: ID админа-автора, таргет-агенты, параметры, подпись, таймстампы, результат. Retention: минимум 1 год на уровне филиала, агрегированные сводки пушатся в HQ.

## Инфраструктура и эксплуатация

### Деплой

Каждый сервер филиала разворачивается через Docker Compose. Стандартные сервисы:
- `postgres` (16-alpine, один volume)
- `server` (umbrella-server FastAPI)
- `nginx` — на хосте (не в compose), используется существующий nginx филиала.

Сеть: umbrella-server торчит портом только на `127.0.0.1:9090`. Nginx проксирует HTTPS к нему.

TLS: Let's Encrypt wildcard через Cloudflare DNS-01 (для филиалов с публичным доменом) или self-signed / внутренний CA (для локальных-only филиалов).

### Мониторинг

Стек Prometheus + Grafana, хостится в HQ.

- Серверы филиалов выставляют `/metrics` (prometheus-формат) — внутренние метрики FastAPI, статистика DB pool, статус синхронизации с HQ.
- Prometheus в HQ скрейпит филиалы через публичные endpoint'ы, защищённые bearer-токеном.
- Grafana-дашборды: здоровье по филиалам, глобальный overview, алерты.
- Alerting: AlertManager → email/Telegram для "филиал offline >10 мин", "высокий error rate", "отставание sync".

Здоровье агентов/spider'ов отслеживается через телеметрию серверов филиалов, а не напрямую Prometheus'ом.

### Обновление агентов

HQ отправляет команды `update_self` как часть координированного rollout'а:

1. HQ-админ выбирает целевые филиалы и версию агента.
2. HQ шлёт "rollout"-команду каждому серверу филиала.
3. Сервер филиала рассылает `update_self` своим агентам пакетами (например, 10 за раз, пауза 1 минута).
4. Агенты скачивают подписанный бинарник с серверной локации филиала или HQ, проверяют подпись, ставят, перезапускаются.
5. Результаты агрегируются обратно в HQ.

Откат: если >X% агентов не обновились за Y минут — rollout прерывается; HQ может отправить обратный `update_self` с предыдущей версией.

### Обновление серверов

Обновления кода самого сервера филиала тоже управляются HQ:

1. HQ ведёт каталог выпущенных версий сервера филиала (теги Docker-образов).
2. HQ шлёт команду "self-update" филиалу.
3. Сервер филиала запускает преднастроенный update-скрипт: `docker compose pull && docker compose up -d` с новым тегом образа.
4. Филиал рапортует новую версию в HQ.

Это означает, что каждому филиалу нужен host-side update-скрипт, вызываемый из контейнера сервера (через shared-socket, SSH-ключ — детали TBD; скорее всего проще всего — host cron проверяет маркер-файл, который сервер пишет).

### Бэкапы

Логические бэкапы PostgreSQL (pg_dump) делаются на каждом филиале ежедневно, хранятся 30 дней локально + еженедельно загружаются во внешнее хранилище (S3-совместимое).

Сервер HQ тоже бэкапится ежедневно, хранится в географически отдельной локации.

## Roadmap доменов

Порядок реализации на сервере филиала:

1. **`instance`** — таблица из одной строки с идентификатором филиала и настройками связи с HQ. Крошечный домен, фундамент для HQ-sync.
2. **`agents`** — CRUD, enrollment (выпуск сертификата), heartbeat.
3. **`groups`** — группировка агентов для таргетинга политик.
4. **`policies`** — правила блокировок с полем `source` (local/global).
5. **`commands` (Фаза 1)** — только безопасные команды: reboot, diagnostics, update_self, apply_config.
6. **`metrics`** — телеметрия агентов, агрегация для HQ.
7. **`events`** — аудит-лог.

**Пилотный деплой** (1–2 филиала, реальные endpoint'ы).

8. **`commands` (Фаза 2)** — download_file, run_installer, переключение сети, именованные скрипты.
9. **`hq-server`** — новое приложение в монорепе.
10. **Sync филиал ↔ HQ** — enrollment, heartbeat, синхронизация политик, маршрутизация команд.
11. **`commands` (Фаза 3)** — кампании, расписание.

Никакая работа над HQ не начинается, пока шаги 1–7 не заработают на реальном филиале.

## Non-goals

Явный список того, что Umbrella делать НЕ пытается. Защита от расползания скоупа.

- **Deep Packet Inspection (DPI).** Фильтрация работает на уровне DNS/URL. Не делаем MITM TLS, не инспектируем содержимое пакетов.
- **Endpoint antivirus / EDR.** Umbrella — не security-агент в смысле AV. Никакого сканирования вредоноса, детекта инъекций процессов.
- **Полноценный MDM.** Никакого mobile device management (политик для Android/iOS). Только desktop-агенты.
- **Произвольное исполнение скриптов на агентах.** Скрипты предрегистрируются и версионируются; никакого `eval(command_string)`.
- **Real-time контроль из HQ.** HQ не имеет прямого управления агентами. Всё идёт через сервер филиала.
- **Free-form multi-tenancy.** Мы обслуживаем одну организацию (с множеством филиалов). Две разные организации = два отдельных деплоя Umbrella. Никакой общей тенантности.
- **Сертификации соответствия.** Не ISO 27001 / SOC 2. Используются разумные практики безопасности, но не регулируемый продукт.

## Открытые вопросы

Пункты, отложенные до будущих решений:

- Точный механизм DNS-блокировки на Windows-агентах (WFP-драйвер? DNS-прокси? гибрид с hosts-файлом?).
- Механизм HQ-аутентификации для корпоративных клиентов (SAML/OIDC?).
- Сроки хранения логов на каждом слое (филиал, HQ, по типам событий).
- Модель approval flow для команд Фазы 2.
- Multi-region деплой HQ (active/passive, read-реплики?). Не проблема, пока масштаб не потребует.
